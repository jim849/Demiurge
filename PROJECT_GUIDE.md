## 1. 项目是什么

一个人工生命 / 进化模拟项目，代号 **Demiurge（造物主计划）**。

它是一个**研究生态与进化动态的实验平台**，不是普通游戏。

两条主线并存：

1. **观察进化**：用尽量少的硬规则，让生态位、克制关系、种群稳态尽可能"自己涌现"，然后观测系统收敛到什么状态。
2. **造物主干预**：我作为"造物主"保留手动干预能力——查看、修改任意个体的基因，甚至凭空创造并投放新个体。

---

## 2. 核心理念

只定义基础物理规则（能量、移动、进食、繁殖、变异、死亡），**不直接编写"聪明"的行为，也不写死"谁吃谁"**。
让生态位、克制关系、稳态形态从个体属性与选择压力中**涌现**出来，然后观测与分析。

研究关注点不只是"谁活到最后"，更是"系统收敛到什么稳态"：稳定共存、捕食者-猎物振荡、一家独大、军备竞赛（红皇后）等。

---

## 3. 世界设定

- **2D 网格世界**，散布植物（基础能量来源）和一群 agent。
- **时间推进**：世界按离散时间步（tick）推进。每个 tick，所有 agent 按**确定的顺序**行动一次。
  更新顺序必须确定，这是"可复现"的前提。
- **克制关系靠涌现，不靠标签**：不用颜色硬性规定谁吃谁。每个 agent 带一组连续的可进化属性
  （体型、速度、食性偏好等），捕食/竞争结果由属性对比自然决定（如体型大的能吃小的）。
  克制关系应从属性中涌现，而非写死。
- **能量**：移动消耗能量、进食补充能量、能量够阈值就繁殖；能量耗尽或被捕食则死亡。
- **繁殖（第一版）**：无性繁殖（分裂），后代基因 = 父代基因 + 随机变异。
  有性繁殖是后续要加入的研究变量，繁殖逻辑须设计得便于将来替换/扩展。

---

## 4. 基因系统（核心数据结构，需重点设计）

这是整个项目的核心数据结构，未来一切扩展都挂在它身上，第一版就要认真设计好。

1. **染色体**：每个 agent 拥有一条"染色体"——一组结构化、有名字的基因（如 体型、速度、食性、繁殖阈值…），
   每个基因带数值和合法取值范围。这条染色体是被遗传、变异、和我手动编辑/创造的对象。

2. **基因型与表现型分离**：
   - 基因（genotype）是抽象数值。
   - 通过一个"映射/表达函数"翻译成实际能力（phenotype，模拟中真正起作用的数值）。
   - 变异只动基因；调平衡只改映射函数。两者解耦。

3. **进化通道**：繁殖时复制父代染色体并施加随机变异，由自然选择决定谁留下。

4. **手动干预通道（务必保留，这是"造物主"的核心能力）**：基因系统要提供干净的接口，支持两件事——
   - (a) 读取和修改任意**已存在**个体的基因；
   - (b) 从一组我指定的基因值**凭空创建**一个全新个体，并投放到世界的指定位置。

   第一版不需要图形编辑/创造界面，但这两个能力（改基因 + 造个体）必须从底层内建，
   哪怕先通过控制台或配置文件实现。漂亮的交互界面是后期的事，接口现在就要留好。

5. **可视化（第一版范围）**：做到"选中一个 agent，能看到它的染色体（基因图谱）"即可。
   整个种群的基因分布直方图、随时间漂移曲线属于数据记录线，先留接口，第一版不实现。

6. **可扩展性**：这条染色体将是项目核心。未来有性繁殖（基因重组）、神经网络
   （权重作为基因挂在染色体上）都要能自然扩展。

---

## 5. Agent 的决策

- 第一版**不上神经网络**，用一个**简单的规则行为**占位（如朝最近的食物移动、随机游走等，具体方案待讨论）。
- **关键**：决策逻辑必须独立成一个**可替换的模块/接口**，将来要整个换成神经网络驱动的决策，
  而 agent 主体和模拟核心不受影响。**不要把决策逻辑焊死在 agent 主体里。**

---

## 6. 第一版目标（不要超纲）

把完整循环跑起来：**能量 → 移动 → 进食 → 繁殖 → 变异 → 死亡**。

完成标准：能 `python main.py` 直接运行，看到一个窗口里 agent 在移动、进食、繁殖，种群随时间进化、属性变化；
能用鼠标选中一个个体查看其染色体；保留手动改基因和凭空造个体的通道（控制台/配置实现即可）。

**结构干净是第一优先级。** 第一版先不要：神经网络、有性繁殖、复杂数据分析、花哨界面。

**规模预期**：第一版大概几百到一两千个体，流畅跑即可。先不追求极致性能，
但数据结构的选择不能将来完全无法扩展（届时可能需要向量化/NumPy）。

## 8. 架构铁律（从第一行代码就遵守，最重要）

1. **逻辑层与渲染层彻底分开**。模拟核心代码里**绝不出现任何 `pygame.` 调用**——核心应搜不到一个 `pygame` 字样。

2. **渲染独立成模块/类**。模拟核心每帧只产出**纯数据**（谁在哪、属性如何、朝向、被选中状态等），
   渲染模块拿数据去画。目标：将来能把 Pygame 渲染模块**整个换成 3D 渲染模块，而核心引擎一行不改**。

3. **位置用向量化方式表示**，不到处散写 x、y，为将来扩展到 3D (x, y, z) 留门。

4. **选中交互的归属**：鼠标点击选中属于渲染/输入层，但"当前选中了哪个个体"是逻辑状态，
   要干净地传递，**别让 Pygame 的点击逻辑渗进核心**。

5. **决策、基因、繁殖、属性系统都要可扩展**，方便后续加入神经网络决策、有性繁殖。

6. **预留数据记录接口**（将来记录各 tick 各类群数量、平均基因等时间序列用于研究稳态）。
   第一版只留口子，不实现分析功能。

7. **可复现**：固定随机种子，同种子跑出同样结果（要求更新顺序确定）。

8. **参数与代码解耦**：所有可调参数集中放在独立配置文件。

9. **跨平台**：文件路径一律用 `pathlib`，不硬编码 `/` 或 `\`；依赖写进 `requirements.txt`；
   不依赖任何系统特定命令。

## 9. 协作方式

- **一步一停**：每次只做一小步，做完停下等我确认再继续，绝不一次性写完整个项目。
- **先讲设计再写代码**：每步动手前，先用文字说明这一步做什么、为什么这样设计，我点头后再写。
- **设计选择交给我拍板**：遇到选择（属性设哪几个、基因用 dict 还是 numpy 数组、
  基因型到表现型怎么映射、网格离散还是连续、决策规则怎么定等），把方案利弊列给我，
  让我决定，不要替我拍板。
- **代码加注释**，关键决策解释清楚。
- **进度同步**：每完成一个里程碑，顺手更新 README，记录当前进度和如何运行（作为实验日志的一部分）。
